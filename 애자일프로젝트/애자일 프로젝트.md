# 애자일 프로젝트 Agile

- 프로젝트 관리자는 조직의 핵심 인력

## 프로젝트의 정의

- 고유한 목표(제품, 서비스, 결과 등)를 산출하기 위해 수행하는 한시적인 노력

## 프로젝트의 특징

- Unique 고유성 : 프로젝트는 고유한 목표에 따라 산출물이 분명한 업무
- Temporary 한시성 : 시작과 끝이 존재
- Uncertain 불확실성 : 표준 프로세스의 반복, 점진적으로 상세화, 

### key point

- 프로젝트화
- 프로젝트관리자의 역량
- 특징
  - 고유하다 Unique
  - 한시적이다 Temporary
  - 불확실하다 Uncertain -> 점진적 상세화 Progressive Falboration
  - 분화를 추진한다 Projects drive change
  - 비즈니스 가치를 창출 한다 Project enable bussness value creation
- 예측형 : 하나의 논리적인 흐름으로 만들어 나감
  애자일 : 변화에 적응함, 상황에따라 범위를 증분하며 프로세스를 공략

## 프로젝트 관리 지식 개요

### 예측형 프로젝트 관리, 폭포수

- 프로젝트 범위 명확
- 요구사항 초기 정의, 고객과 경영진에게 산출물을 체계적으로 문서화하여 보고
- 프로젝트의 범위, 일정, 자원, 비용을 통합 관리

### 적응형 프로젝트 관리, 애자일

- 프로젝트 범위 변경가능성, 불확실성이 큼, 창의적인 구현 필요
- 요구사항을 계속추가하며, 문서격식주의 최소화, 개발자 중심 프로젝트 진행

### xp

- 개발자의 능력에 의존

- 고객과 함께 커뮤니케이션하며 루프를 반복함

  

### Scrum

- 가장 실무적이고 현실적임
- 보편적인 팀워크, 증분
- 복잡한 제품개발을 목표로하는 프레임워크



### 예측형과 애자일 공통점

- 요구사항 식별
- 팀워크로 상호협력적, 이해관계자들 간에 소통
- 범위, 일정, 비용, 자원, 품질, 리스크 등 조건을 균형있게 사용



### 애자일 지식의 발전 과정

#### 성공적인 프로젝트 관리를 위한 3요소

- 프로젝트 관리 방법론 : 전략, 전술
- 프로젝트 관리 전문 소프트웨어 : 무기
- 인적 자원 (유능한 인재와 조직) : 군대



## 애자일 프로젝트 개요

- 경량 방법론을 통칭함
  - Scurm 등
- 애자일의 정의
  - 신속하고, 변화에 유연하며 적응적인 개발을 목표로 하는 다양한 경량 개발 방법론 전체를 일컫는 총칭
  - 증분 : 여러 하위 프로젝트를 통해 점진적으로 프로젝트 범위와 일정을 개발, 산출물을 빠르게 인도
  - 반복 : 단기 단위를 채용해 리스크를 최소화 함
- 애자일의 오해
  - 만능해결책이 아님
  - 문서 격식 주의가 적음
  - 애자일의 표준은 없음
  - 각 좆기에 맞게 테일러링 해야 함
  - 명쾌한 프로세스를 제시하지 않음
  - 애자일 선언문을 통해 공포된 4대 가치 12대 원칙을 중심으로 마음가짐을 갖추는 것이 중요함
- 애자일의 기대 효과
  - 요구사항 능동 대응 가능
  - 빠르고 지속적인 고객 피드백
  - 각 담당자의 권한 강화
  - 진척상황 가시성과 영향력으로 조기에 문제 감지
  - 여러번의 증분 인도 방식으로 낭비 감소
- 애자일 프로젝트의 특징
  - 요구사항과 기대사항을 구분 : 고객은 자기가 진짜 원하는 것을 올바로 제시하지 못함
  - IT 프로젝트 = 팀플레이
  - T자형 인재 : 1인 다역
  - 자기조직화 팀
  - 산출물 관리 : 애자일에서도 산출물관리는 중요, 상황을 정확하고 시각화해 공유
  - 요구사항 정의는 프로젝트 팀이 주도
  - IT프로젝트에서 가장 중요한 것은 리스크 관리 역량 : 목표에 영향을 주는 변수 파악 및 관리, 프로젝트 리스크 관리 계획, 리스크 대응 계획, 리스크보고서와 대시보드 관리 및 예방조치 비상조치 실시
  - 과거의 선례정보를 보고 포기하는 것이 아닌 경험을 통해 극복할 것
  - 프로젝트를 짧게만들어라
  - 애자일의 우선 관리대상
    - 1차 목표 : 프로젝트 리스크 관리 - 반복
    - 2차 목표 : 프로젝트 일정 관리 - 증분

## 애자일 선언문

## 애자일 4대 가치

- 프로스세스나 도구보단 **개인존중과 상호작용**
- 포괄적인 문서보다는 **작동할 수 있는 소프트웨어**
- 계약에 대한 협상보다 **고객과의 협업**
- 계획을 고수하기 보다 **변화에 대응**

## 애자일 12대 원칙

- 우선순위(중요도) 순으로 개발후 인도하여 만족시킴
- 개발 후반부일지라도 요구사항 변경을 환영하라
- 작동하는 소프트웨어를 자주 전달하라, 간격이 짧을 수록 좋음(2주 ~ 2개월)
- 개발자들 끼리는 Daily Scrum 매일 함께 일해야 함
- 동기부여 된 개인들을 중심으로 프로젝트 구성
- 정보전달을 위해서 얼굴을 보고 대화하는 것이 가장 효과적
- 작동하는 소프트웨어가 진척 측정의 주된 척도!
- 지속가능한 개발을 장려
- 기술적 탁월함과 좋은 설계에 대한 지속적 관심이 기민함을 향상 시킴
- 간결함 : 하지 않아도 되는 일을 하지 않는 기술
- 최상의 아키텍처, 요구사항, 설계는 자기조직화 되어 있는 팀에서 나옴
- 팀 차원에서 효과성을 고려하는 것은 과거를 보고 자신의 행동을 조율하고 조정해야 함



## 프로젝트 라이프 사이클

### 프로젝트 라이프 사이클 기초

- PMBOK (Project Life Cycle)
- 마일스톤 | 단계 게이트 : 선행 단계의 문제점 해결 후 다음단계로 진행
- 애자일 프로젝트의 라이프 사이클
  - 시기별 여러 증분범위 제시
  - 증분의 하위에서 **스프린트**라는 더짧은 관리 주기를 서정해 계획 실행 반복
  - 감시와 통제는 전 시기에 걸쳐 실시
- 일반 라이프 사이클
  - 프로젝트의 후반부로 갈수록 리스크 감소 전략 및 활동 배치
  - 변경은 후반부보다 초반부에서 변경(비용문제)
  - 후반 증분단계 에서는 전체 영향도가커 자제함
  - S Curve : 진도율, 작업량, 비용은 누적으로 검토하여 관리해야 함
- 예측형 vs 적응형 프로젝트 관리
  - 예측형 특징
    - 매우 체계적인 계획 수립
    - 변경의 최소화
    - 프로세스 중심
    - 한번의 인도
    - 요구사항 고정
    - 원가 관리
  - 적응형 특징
    - 사람(개발자) 중심
    - 자기조직화 팀  T자형 인재 
    - 반복형 특징
      - 한번의 인도
      - 리스크관리에 초점
      - 요구사항 역동적 : 프로토타입을 통한 피드백&구체화
      - 솔루션의 정확한 개발
      - 스토리보드 기법 : 연속적인 이미지를 통해 전환 순서를 보여줌
      - 프로토타입 : 요구사항수집, 리스크 대응 계획 수립, 충분한 프로토타입 주기
        - 주의사항 : 비현실적인 일정 과기능 기대, 내용보다 외형 검토, 프로토타입에 많은 시간 낭비
    - 증분형 특징
      - 여러번의 더 작은 인도
      - 빠른 속도가 중요
      - 기능을 계속 추가 반복과정을 통해 인도물이 산출됨
      - 우선순위가 높은 기능을 우선으로 개발
    - 애자일 특징
      - 반복형 + 증분형
        - 증분 : 작은 사이클 반복, 최소 기능제품을 진화 시킴
        - 반복기반 : Time Box 내에서 진행
        - 플로우기반 : 요구사항에 따라 Time Box 다르게 설정, 및 과도하게 길어지지 않도록 WIP Limit 설정
      - 중요도 우선 개발, 증분의 반복, 빠른 작은 인도
      - 1차 증분으로 MVP에 해당하는 기능을 개발하고 2차 3차 4차는 그다음 사용자의 관심과 우선순위에 따라 결정
      - 여러번의 인도와 피드백을 통한 고객가치 제공
      - 하이브리드 라이프 사이클
        - 예측형 + 적응형
          - 하이브리드 사이클 유형 1 : 가장 기본적이고 순수한 형태
            - 프로젝트 전반부 애자일, 후반부 예측형으로 진행
            - 프로젝트 개발 범위 불확실성, 복잡성, 리스크 존재시 활용
          - 하이브리드 라이프 사이클 유형2
            - 하나의 프로젝트에서 애자일과 예측형을 동시에 접근함
            - 애자일 : 짧은반복, 일일 기립 회의, 회고
            - 예측형 : 하향식 작업량 산정과 원가 산정, 작업 배정, 진행상황 추적과 보고
          - 하이브리드 라이프 사이클 유형3
            - 일부 범위추가의 불확실성, 복잡성, 리스크가 존재하는 특정 시기에만 애자일 접근
            - 프로젝트의 대부분 작업은 예측형 접근법으로 진행
          - 하이브리드 라이프 사이클 유형4
            - 애자일 접근으로만 진행시 협상불가이거나 실행불가의 경우 유형4 진행
            - 프로젝트 대부분을 애자일 접근법으로 진행
            - 외주로 진행하는 하위프로젝트를 발주자의 관점으로 예측형 접근법 사용
            - 타 벤더 외주 제품이나 서비스를 프로젝트의 본제품과 통합하는 프로젝트
        - 예측형은 프로젝트의 일부 또는 전체 기간 중 다음과 같은 경우에 필요하다.
          \1. 요구사항이 확실히 정해져 있거나 구체적으로 인식한 프로젝트의 범위가 있을 때
          \2. 상호 협력적 개발이 어려울 때
          \3. 프로젝트 외부 조직에서 프로젝트에 참여하는 단계

## 애자일 환경

- 애자일 하지 못한 환경
  - 비유연성 : 방법론, 공식 룰활용
  - 애자일하지 못한 물리적 환경
- 애자일의 요소
  - 최소화된 문서 격식주의 : 적당한 문서화와 재능 의존의 조화
  - 지속가능한 페이스 : 열심히 일함, 개인의 한계 인식, 장기간 작업지속시 페이스 조절
  - 동일 장소 배치 : 팀은 하나의 프로젝트 참여, 가상의 공동 작업
  - 정보의 소통이 원활한 작업 환경 : 즉각적인 상황 공유
  - 조용한 환경
    - 줄여야 할것 : 회사의 간섭, 자질구레한 의무, 관련 없는 작업, 행정적 자업, 팀외부로부터 사내정치
    - 늘려야 할것 : 동기부여, 작업 수행 역량, 가용성
  - 협업을 위한 좌석 배치
    - 의사소통, 협업 작업 환경
    - 팀원은 원하는 자리에 앉을 수 있어야 함
    - 동굴과 공동구역을 제공

# 스크럼

 ## 스크럼 소개

- 문제점을 축적해 추후에 도움이 됨
- 스크럼이란 : 프로젝트를 위한 **상호협력적**, **점진적인** 개발 방법론
- 스크럼은 자기 조직화 팀으로 진행됨 : 서로를 존중하며 협력하고 협업하며 팀전체의 역량강화, 팀단위 생각, 행동, 원활한 의사소통을 위해 구분 없는 열린 공간 유지
- 핵심
  - 가장 많이 사용되는 애자일 기법
  - 1~4주 스프린트
  - 3가지 역할 : 스크럼팀은 3~9명으로 구성된 작은 팀
    - 제품책임자 : 개발할 주체 결정, 제품 소유자
    - 개발팀 : 개발자
    - 스크럼마스터 : 스크럼 프로세스 촉진, 자기조직화팀 구축을 위한 도움, 외부간섭으로 부터 팀 보호, 스크럼 팀에서 제품 책임자(Product Owner)는 가장 가치 있는 소프트웨어를 개발하기 위하여 제품에 필요한 기능을 정의한다. 제품 책임자는 기능을 제품 백로그에 추가하고, 삭제하며, 기능에 우선순위(Priority)를 부여한다.
      이에 비해 스크럼 마스터는 전체를 지원하고 관리하는 사람이다. 스크럼 마스터는 스크럼 프로세스의 촉진자 역할을 한다.
  - 3가지 아티팩트
    - 증분 : 인도할 수 있는 제품 결과
    - 제품 백로그 : 프로젝트 전반에 걸쳐 업데이트 되는 프로젝트 데이터
    - 스프린트 백로그 : 짧은 관리주기에서 갱신되는 제품 범위
  - 5가지 이벤트
    - 스프린트 : 짧은 관리, 보고 주기, 일반적으로 2주, 스프린트 백로그가 계속 추가되고 발전됨

## 스크럼 프로세스

- 스프린트를 정한 후에 제품 백로그를 프로젝트 전반에 걸쳐 업데이트하고 제품을 개발하는 애자일 프로젝트 관리 방법중 하나
- 스크럼은 증분과 반복에 기반한 프로젝트 관리 방법론
  - 증분 = 빌드 = 버전
  - 반복 = 표준 프로세스 = 단계
- 스크럼은 증분과 반복으로 작게 정의하여 수많은 작은 실패를 반복함
- 스크럼은 증분 기간 내에 여러번 시행착오를 할 수 있는 스프린트를 반복함

- 제품 비전
- 제품 로드맵
- 증분 : 고객에게 인도할 수 잇는 결과물, 마일스톤 개념
- 제품 백로그 : 제품 개발을 위해 팀이 유지해 나가는 사용자 중심의 요구사항에 관한 지시 목록
- 착수 회의 : 프로젝트 목적과 개요 설명, 팀원 소개, 현황 검토, 문제영역 식별, 
- 개인과 그룹의 역할과 책임 정의
- 자발적 헌신 의지 도출
- 동기 부여
